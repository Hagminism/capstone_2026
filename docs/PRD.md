# PRD

## 1. 문서 개요

- **프로젝트명:** 통합형 예약 관리 시스템
- **문서 목적:** 팀이 동일한 제품 목표, MVP 범위, 핵심 사용자 흐름, 기술 방향을 공유할 수 있도록 제품 요구사항을 정의한다.
- **대상 독자:** 프론트엔드, 백엔드, 기획, 발표 자료 작성 담당자
- **문서 기준일:** 2026-03-23

---

## 2. 제품 비전

여러 업종에 흩어진 예약 경험을 하나의 앱으로 통합한다. 사용자는 위치 기반으로 주변 업장을 탐색하고, 업종별로 다른 예약 대상을 공통된 방식으로 조회하고 예약할 수 있어야 한다. 관리자는 자신의 업장 정보와 예약 가능 자원을 관리하고, 실시간으로 예약 현황을 확인할 수 있어야 한다.

이 프로젝트는 예약 대상을 `Resource`로 추상화하여 식당 좌석, 카페 테이블, 미용 서비스, 스터디카페 좌석 등 서로 다른 예약 단위를 하나의 도메인 모델로 처리하는 것을 핵심 가치로 삼는다.

---

## 3. 해결하려는 문제

- 사용자는 업종마다 서로 다른 앱과 예약 흐름을 사용해야 한다.
- 업주 입장에서는 예약 관리, 정보 노출, 고객 응대를 여러 채널에서 분산 관리해야 한다.
- 예약 도메인은 시간, 인원, 자원, 운영 시간, 중복 예약 방지 등 정합성이 중요한데, 이를 범용적으로 처리하는 구조가 필요하다.

---

## 4. 제품 목표

### 4.1 사용자 목표

- 현재 위치를 기준으로 주변 업장을 빠르게 탐색한다.
- 원하는 조건에 맞는 업장을 검색하고 예약 가능 여부를 확인한다.
- 날짜, 시간, 인원 조건에 맞게 예약을 생성하고 내 예약 내역을 확인한다.

### 4.2 관리자 목표

- 업장 인증 후 자신의 업장 정보를 등록하고 수정한다.
- 업장에 속한 예약 가능 자원을 관리한다.
- 들어온 예약을 확인하고 운영에 활용한다.

### 4.3 비즈니스 목표

- 업종 비종속적인 예약 플랫폼의 기반을 구축한다.
- 추후 AI 추천, 외부 예약 플랫폼 연동, 쿠폰/스탬프 기능을 확장 가능한 구조로 준비한다.

---

## 5. 대상 사용자

### 5.1 일반 사용자

- 주변 식당, 카페, 미용실, 스터디카페 등을 검색하고 예약하려는 사용자
- 여러 예약 앱을 따로 설치하고 싶지 않은 사용자

### 5.2 관리자(업장 운영자)

- 자신의 업장을 앱에 등록하고 예약을 관리하려는 사용자
- 업장 정보, 운영 시간, 예약 가능 자원을 직접 설정하려는 사용자

### 5.3 플랫폼 관리자

- 관리자 회원가입 승인 및 사업자 인증 보조 검증을 수행하는 운영자

## 6. MVP 범위

### 6.1 MVP에 포함

- 사용자 인증
  - 일반 사용자 회원가입/로그인
  - 관리자 회원가입/로그인
  - Firebase Authentication 기반 인증 처리
- 업장 탐색
  - 홈 화면 진입
  - 검색 페이지
  - 지도 기반 주변 업장 조회
  - 카테고리, 별점, 운영 시간 기반 필터
- 업장 상세
  - 업장 기본 정보 조회
  - 사진, 메뉴, 리뷰, 매장 정보 탭 제공
  - 북마크(저장) 기능
- 예약
  - 날짜, 시간, 인원 선택
  - 예약 생성
  - 예약 내역 조회
  - 중복 예약 방지
- 관리자 기능
  - 사업자 정보 입력
  - 사업자등록증 첨부
  - 업장 정보 등록/수정
  - 예약 현황 조회
- 마이페이지
  - 프로필 수정
  - 예약/리뷰 내역 조회
  - 공지사항, 약관, 문의 진입
- 알림
  - 주요 예약 관련 알림 목록 조회

### 6.2 MVP에서 제외

- AI 기반 자연어 추천의 정교한 대화형 큐레이션
- 외부 예약 플랫폼과의 실시간 재고 동기화
- 쿠폰/스탬프 자동 적립
- 숙소 예약 도메인 확장
- 결제 수단 저장 및 결제 연동

### 6.3 MVP 이후 확장 대상

- AI 추천
- 외부 플랫폼 딥링크/애그리게이터
- 쿠폰/멤버십
- 업종별 세부 예약 규칙 고도화

## 7. 핵심 사용자 시나리오

### 7.1 일반 사용자 예약 흐름

1. 사용자가 앱에 로그인한다.
2. 홈 또는 지도 화면에서 주변 업장을 탐색한다.
3. 검색어 또는 필터를 이용해 업장을 좁힌다.
4. 업장 상세 페이지에서 운영 시간, 리뷰, 예약 가능 정보를 확인한다.
5. 예약하기 버튼을 눌러 날짜, 시간, 인원을 선택한다.
6. 예약을 완료하고 내역 및 알림에서 결과를 확인한다.

### 7.2 관리자 등록 흐름

1. 관리자가 회원가입 시 관리자 유형을 선택한다.
2. 업장 정보와 사업자 정보를 입력한다.
3. 사업자등록증을 첨부한다.
4. 플랫폼 또는 검증 로직을 통해 승인 절차를 진행한다.
5. 승인 후 업장과 예약 자원을 등록한다.
6. 예약 발생 시 관리자 화면에서 현황을 조회한다.

## 8. 기능 요구사항

### 8.1 인증

- 일반 사용자와 관리자를 구분하여 가입/로그인할 수 있어야 한다.
- 이메일 로그인 또는 소셜 로그인을 지원할 수 있어야 한다.
- 인증은 Firebase Authentication을 사용한다.
- 관리자의 경우 사업자 관련 추가 정보를 입력해야 한다.

### 8.2 홈

- 검색 바로 업장을 탐색할 수 있어야 한다.
- 카테고리별 업장 진입이 가능해야 한다.
- 알림 진입 아이콘이 제공되어야 한다.

### 8.3 검색

- 텍스트 기반 업장 검색이 가능해야 한다.
- 검색 결과 목록에서 업장명, 사진, 주소를 제공해야 한다.
- 업장 선택 시 상세 화면으로 이동해야 한다.

### 8.4 지도

- 현재 위치를 기준으로 주변 업장을 표시해야 한다.
- 반경 재검색이 가능해야 한다.
- 업종, 별점, 운영 시간 등의 필터를 지원해야 한다.

### 8.5 업장 상세

- 사진, 업장명, 업종, 별점, 리뷰 수, 운영 시간 정보를 보여줘야 한다.
- 메뉴, 사진, 리뷰, 매장 정보 탭을 제공해야 한다.
- 저장 및 예약 진입 기능이 있어야 한다.

### 8.6 예약

- 날짜, 시간, 인원을 선택할 수 있어야 한다.
- 업장별 자원과 운영 규칙에 맞는 예약 가능 여부를 보여줘야 한다.
- 예약 생성 시 같은 시간대 자원 중복 할당이 방지되어야 한다.

### 8.7 관리자

- 업장 기본 정보와 운영 시간을 관리할 수 있어야 한다.
- 예약 가능 자원을 등록하고 수정할 수 있어야 한다.
- 예약 현황을 확인할 수 있어야 한다.

### 8.8 마이페이지 및 알림

- 프로필 정보 수정이 가능해야 한다.
- 예약 내역 및 리뷰 내역 확인이 가능해야 한다.
- 알림 목록 조회가 가능해야 한다.

## 9. 비기능 요구사항

### 9.1 데이터 정합성

- 예약 생성/변경/취소 과정에서 데이터 충돌을 방지해야 한다.
- 동일 자원에 대한 중복 예약이 발생하지 않아야 한다.

### 9.2 권한 및 보안

- 일반 사용자, 관리자, 플랫폼 관리자 권한이 분리되어야 한다.
- 업장 데이터는 멀티테넌트 구조로 격리되어야 한다.

### 9.3 확장성

- 새로운 업종의 예약 자원을 추가해도 기존 구조를 크게 바꾸지 않아야 한다.
- AI 추천 및 외부 연동 기능을 후속 확장할 수 있어야 한다.

### 9.4 운영 안정성

- 인증, 예약, 업장 조회 등 주요 흐름에서 장애 원인을 추적할 수 있어야 한다.
- 크래시 및 주요 이벤트를 수집할 수 있어야 한다.

## 10. 기술 방향

### 10.1 앱 구조

- Flutter 기반 크로스플랫폼 앱으로 개발한다.
- MVVM 기반 Clean Architecture를 적용한다.
- 유저 액션 처리 방식은 MVI 성격을 반영한다.
- `go_router`, `get_it`, `ChangeNotifier`, `freezed`, `build_runner`를 사용한다.

### 10.2 백엔드 구조

- 인증 및 부가 기능은 Firebase를 사용한다.
  - Firebase Authentication
  - FCM
  - Analytics
  - Crashlytics
- 핵심 데이터 저장은 Supabase를 사용한다.
  - PostgreSQL 기반 RDBMS
  - 예약, 업장, 자원, 리뷰 등 정합성이 중요한 데이터 저장

### 10.3 기술 선택 이유

- 예약 시스템은 시간, 자원, 상태 전이, 중복 방지 등 관계형 데이터 모델이 적합하다.
- Firebase Authentication은 로그인 및 사용자 인증 흐름을 빠르게 구축하기 좋다.
- Supabase는 RDBMS 기반으로 예약 도메인의 정합성, 검색, 권한 정책 설계에 유리하다.

## 11. 핵심 도메인 모델

### 11.1 주요 엔티티

- `User`
  - 일반 사용자 또는 관리자 역할
- `Store`
  - 업장 기본 정보, 위치, 운영 시간, 업종
- `Resource`
  - 예약 가능한 대상
  - 예: 좌석, 테이블, 서비스 슬롯
- `Reservation`
  - 사용자, 업장, 자원, 날짜, 시간, 인원, 상태
- `Review`
  - 업장 평가 및 후기
- `Notification`
  - 예약 생성/변경/취소 등의 알림
- `OwnerVerification`
  - 사업자 인증 및 승인 상태

### 11.2 핵심 도메인 원칙

- 모든 예약 대상은 `Resource`로 추상화한다.
- 예약 가능 여부는 업장 운영 시간, 자원 상태, 시간 슬롯 규칙을 함께 고려해 계산한다.
- 관리자마다 자신의 업장 데이터에만 접근할 수 있어야 한다.

## 12. 성공 지표

### 12.1 MVP 완료 기준

- 사용자가 회원가입 후 주변 업장을 검색하고 예약까지 완료할 수 있다.
- 관리자가 업장을 등록하고 예약 현황을 조회할 수 있다.
- 예약 중복 방지와 권한 분리가 정상적으로 동작한다.

### 12.2 추후 측정 지표

- 회원가입 완료율
- 검색 후 상세 페이지 진입률
- 상세 페이지 대비 예약 완료율
- 예약 실패율
- 관리자 승인 완료율

## 13. 리스크 및 대응

### 13.1 외부 API 의존성

- 국세청 API 장애 또는 지연 시 관리자 승인 흐름이 막힐 수 있다.
- 대응: 수동 승인 절차를 fallback으로 둔다.

### 13.2 AI 환각

- AI 추천이 실제 운영 정보와 다른 결과를 제시할 수 있다.
- 대응: AI 추천은 MVP 제외, 추후 도입 시 DB 검증 단계를 강제한다.

### 13.3 외부 예약 플랫폼 제약

- 외부 플랫폼의 실시간 데이터 접근이 어려울 수 있다.
- 대응: 초기에는 딥링크 또는 웹뷰 이동 방식으로 제한한다.

### 13.4 동시성 문제

- 같은 시간대 같은 자원에 중복 예약이 발생할 수 있다.
- 대응: Supabase/PostgreSQL 트랜잭션과 제약조건 기반으로 처리한다.
