# 프로젝트 제안서 초본 : 통합형 예약 관리 시스템

## 1. 프로젝트 수행 목적

### 1.1 프로젝트 정의

모든 업종에 범용적으로 적용 가능한 추상화 기반의 통합형 예약 관리 시스템 및 AI 기반 맞춤형 추천 기능을 탑재한 예약 허브 크로스플랫폼 어플리케이션 개발.

### 1.2 프로젝트 배경

최근 식당, 카페, 미용실 등 다양한 서비스 분야에서 예약 시스템 도입이 일반화됨에 따라 사용자의 편의성은 향상되었으나, 동시에 방문하는 업장마다 서로 다른 플랫폼을 개별적으로 이용해야 하는 서비스의 파편화 문제가 발생하고 있다.

이러한 환경은 사용자에게 앱 설치 및 회원가입의 피로도를 유발할 뿐만 아니라, 예약 데이터가 각기 다른 채널로 분산되어 통합적인 관리를 어렵게 만든다.

우리는 이러한 비효율을 개선하기 위해 업종의 경계를 넘어 예약 가능한 모든 자원을 '객체(Resource)'로 정의하고 관리하는 범용 예약 엔진을 구축하고자 한다.

본 프로젝트를 통해 구현될 통합 인터페이스는 특정 분야에 국한되지 않고 예약이 필요한 모든 산업군에 유연하게 적용될 수 있으며, 궁극적으로는 사용자에게 단일화된 예약 경험을 제공하고 점주에게는 데이터 기반의 스마트한 매장 운영 환경을 지원하여 서비스 만족도를 극대화할 수 있을 것이다.

### 1.3 프로젝트 목표

- **가. 통합 예약 인터페이스 제공**
    - 요식업, 카페, 미용실 등 업종에 구애받지 않고 '예약 가능한 자원'이 있는 곳이라면 어디든 적용 가능한 예약 시스템 구축.
- **나. 위치 기반 맞춤형 탐색**
    - 내 주변 매장 검색 및 업종, 별점, 운영 시간 기반의 필터링 기능 제공.
- **다. 추상화 기반 예약 모델링**
    - 공간, 사물, 서비스 등 예약 가능한 모든 대상을 '자원(Resource)'으로 객체화하여 유연한 예약 규칙(시간, 인원 단위 등) 적용.
- **라. AI 기반 지능형 업장 추천**
    - 사용자의 자연어 요구사항(인원, 장소, 목적 등)을 분석하여 최적의 업장을 제안하는 대화형 인터페이스 구현.
- **마. 외부 예약 플랫폼 연동 (Aggregator)**
    - 자체 시스템뿐만 아니라 네이버 예약, 캐치테이블, 테이블링 등 외부 플랫폼과의 연동을 통해 사용자에게 통합된 예약 선택권 제공.

## 2. 프로젝트 개요

### 2.1 프로젝트 설명

- 본 프로젝트는 **Flutter** 프레임워크를 활용하여 사용자의 현 위치를 기반으로 주변 업장의 예약 가능 자원을 실시간으로 탐색하고 예약을 수행하는 크로스 플랫폼 모바일 애플리케이션을 구현한다.
- LBS(위치 기반 서비스) 기술을 통해 반경 내 업장 정보를 지도상에 시각화하고, Firebase의 실시간 데이터베이스를 활용하여 업종별(식당, 카페, 미용실 등)로 상이한 예약 규칙과 잔여 자원 현황을 즉각적으로 동기화하는 기능을 제공한다.
- 또한, 국세청 API 연동을 통한 사업자 인증 체계와 다중 테넌트 아키텍처를 적용하여, 다수의 관리자가 독립적으로 매장 레이아웃을 설정하고 예약을 관리할 수 있는 통합 운영 환경을 구축한다.

### 2.2 프로젝트 구조

- **Client (App)**
    - 일반 사용자용(예약, 결제, AI 큐레이션 채팅, 검색) 및 관리자용(매장 정보 등록, 예약 승인/관리) 인터페이스.
- **Server**
    - 사용자 인증, 실시간 예약 상태 동기화 처리, 다중 테넌트(Multi-tenancy) 환경 데이터 격리.
    - AI 프롬프트 엔지니어링: LLM API와 통신하여 사용자 의도 분석 및 추천 로직 수행.
- **DB**
    - 사용자 정보, 매장 데이터(사진, 리뷰, 레이아웃), 객체화된 자원 데이터베이스 구축.
- **API 연동**
    - 국세청 사업자등록정보 진위확인 API 연동, 소셜 로그인 API 연동, LLM API 연동.

### 2.3 시나리오

- **가. 사용자 인증 및 가입**
    - 일반 사용자는 소셜 로그인으로 가입하고, 관리자는 국세청 API 및 사업자등록증 첨부를 통해 업장 소유주 인증을 거친다.
- **나. 위치 기반 검색 및 예약**
    - 사용자가 앱 실행 시 현 위치 기반 일정 반경 내의 업장들을 지도에 핀으로 확인.
    - 필터(업종, 별점 등)를 적용하여 원하는 업장 선택.
    - 업장 상세 페이지에서 실시간 예약 가능 여부를 확인하고 인원/시간을 선택하여 예약.
- **다. 실시간 예약 동기화**
    - 사용자가 예약을 완료하면 관리자 화면에 즉각적으로 상태가 반영되어 중복 예약을 방지한다.
- **라. AI 맞춤형 큐레이션**
    - 사용자가 "내일 10시 혜화 5명 회식 장소"라고 입력 시, AI가 위치, 운영 시간, 메뉴, 리뷰 데이터를 분석하여 상위 3개 업장을 추천하고 즉시 예약 화면으로 연결한다.
- **마. 하이브리드 예약 확인**
    - 특정 업장 선택 시 본 플랫폼의 예약 현황과 더불어 "네이버 예약으로 확인하기" 버튼을 제공하여 타 플랫폼과의 실시간 비교 및 전환 예약을 지원한다.

### 2.4 기대효과

특정 업종에 매몰되지 않고 '예약'이 필요한 모든 소상공인 시장으로 확장 가능하여, 사용자의 앱 설치 피로도를 줄이고 점주의 플랫폼 관리 리소스를 획기적으로 절감할 수 있다.

### 2.5 제약조건

- **가. 비회원 제한**
    - 미성년자 출입 가능 시간을 제한하는 업장, 주류 판매 업장 등의 경우 예약 페이지 진입 단계에서 강제적인 인증 절차가 필요하다.
- **나. API 의존성**
    - 국세청 사업자 정보 진위 확인 API 서버의 상태에 따라 관리자 회원가입 및 승인 절차에 지연이 발생할 수 있다.
- **다. LLM 엔진의 응답 무결성**
    - AI 기반 추천 기능 사용 시, 모델의 할루시네이션(Hallucination, 환각 현상)으로 인해 영업시간이나 메뉴 정보가 실제와 다르게 생성될 가능성이 존재하므로, 최종 예약 전 반드시 실제 DB 데이터와의 대조 과정을 거쳐야 한다.
- **라. 외부 플랫폼 데이터 접근 제한**
    - 타 예약 서비스의 보안 정책 및 API 공개 여부에 따라 실시간 잔여석 조회가 제한될 수 있으며, 이 경우 딥링크(Deep Link) 또는 웹뷰를 통한 사용자 이동 방식으로 보완한다.

### 2.6 관련기술

- **다중 테넌트(Multi-tenancy) 아키텍처**
    - 단일 소프트웨어 인스턴스가 여러 사용자 그룹(Tenants)에게 서비스를 제공하면서도, 각 그룹의 데이터와 설정이 서로 물리적 또는 논리적으로 격리되는 아키텍처 모델이다.
    - 본 프로젝트에서는 다수의 개별 업주(관리자)가 동일한 플랫폼에 입점하더라도 서로의 예약 정보나 매출 데이터에 접근할 수 없도록 엄격한 데이터 보안 및 독립적 환경을 구축하는 데 사용된다.
    - 이는 대규모 SaaS(Software as a Service) 모델의 핵심적인 설계 기법이다.
- **LBS (Location Based Service)**
    - LBS(Location Based Service)는 휴대폰이나 GPS 등 기기의 위치 정보를 활용하여 사용자에게 필요한 정보를 제공하는 시스템이다.
    - 본 프로젝트에서는 사용자의 위도와 경도 좌표를 실시간으로 수집하고, 이를 기반으로 설정된 반경 내의 등록 업장을 검색 및 쿼리(Query)하여 지도상에 매핑하는 핵심 기술로 활용된다.
    - 단순히 지도를 표시하는 것을 넘어, 사용자의 이동에 따른 실시간 재검색과 거리 기반의 필터링 기능을 구현하기 위한 기반 기술이다.
- **LLM 기반 시맨틱 검색**
    - 사용자가 입력한 자연어(예: "혜화역 근처 5명 회식하기 좋은 돼지고기 맛집")에서 의도(Intent)와 엔티티(Entity)를 추출하여, DB 내의 업장 데이터와 매칭하거나 외부 정보를 탐색하여 최적의 결과를 도출하는 기술이다.
- **API 애그리게이터 (Aggregator)**
    - 서로 다른 플랫폼의 API를 하나로 통합하여 제공하는 기술이다.
    - 본 프로젝트에서는 자체 예약 데이터와 외부 플랫폼(네이버 등)의 예약 가능 여부를 대조하고 사용자에게 통합된 인터페이스를 제공하는 데 활용된다.

### 2.7 개발도구

- **Android Studio**
    - 안드로이드는 휴대전화를 비롯한 장치를 위한 운영 체제와 미들웨어, 사용자 인터페이스 그리고 표준 응용 프로그램을 포함하고 있는 소프트웨어 스택이다.
    - 안드로이드 스튜디오는 이러한 안드로이드 앱 개발을 위한 공식 통합 개발 환경(IDE)으로, 코드 편집, 디버깅, 성능 분석 도구 및 에뮬레이터를 제공한다.
    - 개발자들이 효율적으로 응용 프로그램을 작성할 수 있도록 소프트웨어 개발 키트(SDK)를 통해 각종 도구들과 응용 프로그램 프로그래밍 인터페이스(API)를 제공하며, 본 프로젝트에서는 Flutter 플러그인을 결합하여 메인 개발 환경으로 사용한다.
- **XCode**
    - 애플(Apple)에서 제공하는 macOS 전용 통합 개발 환경(IDE)으로, iOS, iPadOS, macOS 등의 소프트웨어를 개발하기 위한 필수 도구이다.
    - 인터페이스 빌더, 자산 관리 도구 및 iOS 시뮬레이터를 포함하고 있으며, Flutter 프로젝트의 iOS 타겟 빌드와 패키징을 담당한다.
    - 특히 기기의 권한 설정 및 배포 환경 구성을 위해 반드시 필요한 도구이므로, 진정한 크로스 플랫폼 서비스를 구현하기 위해 Android Studio와 병행하여 사용할 예정이다.
- **Flutter**
    - 구글(Google)에서 출시한 오픈 소스 UI 소프트웨어 개발 키트(SDK)로, 하나의 코드 베이스로 안드로이드와 iOS 앱을 동시에 제작할 수 있는 크로스 플랫폼 프레임워크이다.
    - 자체 렌더링 엔진인 Skia를 사용하여 모든 플랫폼에서 일관되고 부드러운 사용자 인터페이스를 제공하며, Dart 언어를 기반으로 컴파일된 바이트코드를 구동한다.
    - 또한 'Hot Reload' 기능을 통해 개발 중 코드 수정 사항을 즉시 확인할 수 있는 효율성을 제공한다.
- **Firebase**
    - 구글(Google)에서 제공하는 모바일 및 웹 애플리케이션 개발 플랫폼으로, 서버 인프라를 직접 구축하지 않고도 백엔드 기능을 구현할 수 있는 BaaS(Backend as a Service) 솔루션이다.
    - NoSQL 기반의 실시간 데이터베이스인 Cloud Firestore와 사용자 인증을 위한 Authentication, 리소스 저장을 위한 Storage 등 다양한 API를 제공한다.
    - 특히 다수의 클라이언트 간 실시간 데이터 동기화가 용이하고 확장성이 뛰어나기 때문에, 본 프로젝트의 핵심인 실시간 예약 상태 관리 및 다중 테넌시 데이터 격리를 구현하는 데 최적의 도구라 생각하여 사용할 예정이다.
- **LLM (Large Language Model) API**
    - 대규모 언어 모델을 기반으로 자연어 처리 및 생성 기능을 제공하는 인터페이스이다.
    - 사용자의 복합적인 예약 요구사항(장소, 시간, 목적 등)을 분석하여 의도를 파악하고, 최적의 업장을 추천하거나 외부 서비스와의 연동 정보를 요약하여 제공하는 지능형 큐레이션 엔진으로 활용한다.

## 3. 프로젝트 추진 체계 및 일정

### 3.1 역할 분담

- **지도교수:** 허준영
- **팀장:** 김동현
- **프론트엔드 개발:** 이학민, 이세상
- **백엔드 개발:** 오성민, 백상준, 김동현

### 3.2 작업 흐름도

1. **요구사항 분석:** 유스케이스 분석, 상세 요구사항 명세.
2. **설계:** 앱 UI/UX 와이어프레임 설계, DB 스키마 및 추상화 자원 모델링.
3. **구현:** 프론트엔드 및 백엔드 연동, 실시간 동기화 구현, 위치 기반 지도 연동.
4. **테스트 및 완료:** 단위 테스트, 실기기 빌드 테스트, 최종 발표.

## 추가 회의 결과

- 서비스 방향성
    - 대기업 어플에서 사용 가능한 기능들을 개인 업장에서도 사용할 수 있게 해줄게!
- 쿠폰(스탬프) 기능도 추가했으면 좋겠다는 의견
    - 카페로 예를 들면, 커피 10잔 마시면 1잔 주는 형태의 스탬프 쿠폰을 나눠주는 가게가 종종 있는데, 쿠폰을 항상 들고 다니진 않으니 스탬프 하나 찍힌 쿠폰 여러 개를 소지하게 되는 경우가 발생.
    - 우리 어플리케이션을 통해 매장을 이용하면 쿠폰 적립/이용을 자동으로 해주면 좋겠다는 의견이었음.
- 여유가 될 경우 숙소에 대해서도 예약이 가능했으면 좋겠다는 의견
- 흔히 쓰는 스터디카페 앱의 좌석 예약은 해당 레이아웃의 좌표 값에 예약 거는 것이라고 한다.
